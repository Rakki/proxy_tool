# Proxy Client MVP

Flutter-приложение-клиент для Android, которое:

- поднимает локальный VPN через `VpnService`,
- перенаправляет трафик только выбранных приложений,
- отправляет этот трафик через upstream proxy,
- поддерживает несколько типов прокси: `SOCKS5`, `HTTP`, `HTTPS CONNECT`,
- оставляет архитектурный запас под расширение на другие протоколы.

## Цель MVP

Собрать рабочую первую версию, в которой пользователь может:

1. выбрать установленные приложения;
2. указать параметры прокси;
3. запустить VPN;
4. направить трафик только выбранных приложений через этот прокси;
5. остановить VPN и получить понятный статус/ошибку.

## Границы проекта

Первая версия целенаправленно ограничена:

- целевая платформа: только Android;
- Flutter используется для UI, хранения конфигурации и orchestration;
- критическая VPN-логика пишется нативно на Android;
- iOS не входит в MVP;
- в первой версии не делать MITM и не пытаться разбирать HTTPS-трафик;
- не писать собственный полный VPN/proxy stack с нуля, если можно встроить готовое ядро.

## Почему не чистый Flutter

Flutter подходит для:

- экранов,
- выбора приложений,
- формы конфигурации,
- логов и статусов,
- локального хранения настроек.

Но выборочная маршрутизация трафика по приложениям требует Android API:

- `VpnService`
- `VpnService.Builder`
- `addAllowedApplication(...)`
- foreground service

Поэтому архитектура будет гибридной: `Flutter UI` + `Kotlin Android layer` + `network engine`.

## Архитектура

### Flutter layer

Отвечает за:

- список приложений;
- экран конфигурации прокси;
- сохранение настроек;
- запуск и остановку VPN;
- отображение статуса;
- вывод пользовательских логов.

### Android native layer

Отвечает за:

- запрос VPN permission;
- поднятие `VpnService`;
- настройку `allowedApplications`;
- lifecycle foreground service;
- связь с сетевым движком;
- системные ошибки и статусы.

### Network engine

Нужен для:

- обработки трафика из TUN;
- отправки его в upstream proxy;
- DNS-логики;
- UDP/QUIC сценариев;
- дальнейшего расширения на новые протоколы.

Предпочтительный путь:

- использовать готовое ядро, например `sing-box`;
- не писать полный стек самостоятельно, пока нет сильной причины.

## Поддерживаемые proxy types

На старте:

- `SOCKS5`
- `HTTP`
- `HTTPS CONNECT`

Позже можно расширить:

- `Shadowsocks`
- `VMess`
- `VLESS`
- `Trojan`

Важно: UI и модель данных должны быть универсальными с первого дня, чтобы добавление нового типа меняло только config mapping, а не всю архитектуру.

## Конфигурационная модель

Нужна одна нормализованная модель конфигурации. Минимальные поля:

- `proxyType`
- `host`
- `port`
- `username`
- `password`
- `useTls`
- `selectedApps`
- `dnsMode`
- `bypassDomains`
- `bypassSubnets`
- `enableUdp`

Дополнительно позже:

- `mtu`
- `ipv6Mode`
- `remoteDns`
- `proxyName`
- `connectTimeoutMs`

Нужен platform channel с такими операциями:

Задача канала: скрыть детали нативной реализации и держать Flutter на уровне команд и статусов.

## Package Visibility And Play Store

Для экрана выбора приложений Android-приложению может потребоваться `QUERY_ALL_PACKAGES`, если нужно показать пользователю полный список установленных приложений.

Для internal/dev сборок это допустимо, но для публикации в Google Play это чувствительное разрешение, и его нельзя объяснять аргументом "так проще".

Обоснование должно быть таким:

- это нужно для основной пользовательской функции приложения;
- приложение реализует per-app proxy/VPN routing;
- пользователь должен иметь возможность выбрать любое установленное приложение;
- заранее неизвестно, какие именно пакеты установлены у конкретного пользователя;
- статический список `<queries>` не решает задачу, потому что выбор приложений динамический;
- список приложений используется только локально для app picker и применения routing rules;
- эти данные не используются для рекламы, аналитики или перепродажи.

Практическая позиция для этого проекта:

- `QUERY_ALL_PACKAGES` допустим для dev/internal use;
- для Google Play approval может потребоваться отдельное обоснование в Permissions Declaration Form;
- ту же функциональность нужно явно описывать в listing, privacy policy и при необходимости внутри приложения;
- лучше держать отдельные flavor'ы:
  - internal/dev build с `QUERY_ALL_PACKAGES`,
  - play build без него или с отдельной стратегией package visibility.

Если проект пойдет в публикацию, отдельно подготовить:

- текст для Permissions Declaration Form;
- текст для Privacy Policy;
- in-app disclosure о том, зачем приложению нужен доступ к списку установленных приложений.

## План реализации

### Шаг 1. Зафиксировать MVP

Сначала утвердить:

- только Android;
- только per-app routing;
- на старте `SOCKS5` и `HTTP/HTTPS CONNECT`;
- одно сетевое ядро;
- без iOS и без desktop.

Результат: понятные рамки, чтобы не расползся объем.

### Шаг 2. Спроектировать data model

Нужно создать:

- модель прокси-конфига;
- модель установленного приложения;
- модель runtime status;
- модель ошибок.

Результат: Flutter UI и native слой говорят на одном языке.

### Шаг 3. Реализовать список приложений

На Android получить список приложений через `PackageManager`, во Flutter:

- показать иконки и названия;
- добавить поиск;
- сохранять выбранные package names.

Результат: готов базовый пользовательский сценарий выбора приложений.

### Шаг 4. Подключить `VpnService`

Сделать минимальный рабочий Android слой:

- `prepare` permission flow;
- `VpnService.Builder`;
- `addAllowedApplication(...)`;
- foreground notification;
- `start` / `stop`.

Результат: поднимается пустой VPN, ограниченный выбранными приложениями.

### Шаг 5. Проверить lifecycle

Проверить отдельно:

- повторный запуск VPN;
- остановку VPN;
- уход приложения в background;
- возврат из background;
- поведение после kill процесса;
- поведение после reboot устройства.

Результат: сервис стабильно живет в реальных условиях.

### Шаг 6. Подключить network engine

Встроить выбранное ядро, предпочтительно `sing-box`, и добиться:

- запуска engine из Android слоя;
- передачи в него runtime config;
- остановки и рестарта без утечек.

Результат: VPN-сервис связан с реальным сетевым движком.

### Шаг 7. Реализовать `SOCKS5`

Начать с `SOCKS5`, потому что это обычно самый прямой путь для MVP.

Нужно добиться:

- успешного TCP-маршрута через SOCKS5 proxy;
- работы авторизации;
- понятных ошибок при недоступности сервера.

Результат: первая рабочая end-to-end версия.

### Шаг 8. Добавить `HTTP` / `HTTPS CONNECT`

После рабочего `SOCKS5` расширить генерацию конфигурации и routing path для:

- `HTTP proxy`
- `HTTPS CONNECT`

Результат: приложение поддерживает два распространенных класса upstream proxy.

### Шаг 9. Довести DNS

Нужно отдельно решить:

- где резолвить домены;
- как исключать DNS leaks;
- как вести себя при remote DNS;
- как обрабатывать IPv4/IPv6.

Результат: приложения работают стабильнее, особенно при реальном повседневном использовании.

### Шаг 10. Добавить UDP

Нужно проверить и реализовать:

- UDP forwarding;
- работу приложений, использующих QUIC;
- fallback, если конкретный proxy type плохо сочетается с UDP.

Результат: меньше сценариев частичной поломки у современных приложений.

### Шаг 11. Добавить bypass rules

Нужно поддержать:

- bypass по доменам;
- bypass по IP/subnet;
- direct mode для локальной сети;
- исключения для служебных адресов.

Результат: приложение становится пригоднее для реального повседневного использования.

### Шаг 12. Логи и диагностика

Добавить:

- статусы `idle`, `starting`, `running`, `stopping`, `error`;
- пользовательские сообщения об ошибках;
- внутренние debug logs;
- экран последних событий.

Результат: проще разбирать проблемы и поддерживать приложение.

### Шаг 13. Безопасность хранения

Нужно отдельно продумать:

- хранение логина/пароля;
- скрытие секретов из логов;
- очистку данных при reset;
- безопасное восстановление профиля после перезапуска.

Результат: конфиги не текут в открытом виде без необходимости.

### Шаг 14. Тестирование

Минимальный план тестов:

- unit-тесты моделей и валидации;
- widget-тесты форм и экранов;
- ручные тесты VPN lifecycle;
- ручные тесты per-app routing на реальных приложениях;
- тесты переключения между `SOCKS5` и `HTTP`;
- тесты ошибок соединения и авторизации.

Результат: базовый контроль качества до расширения функциональности.

## Порядок работ

Рекомендуемая последовательность:

1. Спроектировать модели данных.
2. Сделать экран списка приложений.
3. Сделать экран конфигурации прокси.
4. Поднять пустой `VpnService`.
5. Добиться стабильного lifecycle.
6. Подключить `sing-box` или другое ядро.
7. Запустить `SOCKS5`.
8. Добавить `HTTP/HTTPS CONNECT`.
9. Довести DNS.
10. Довести UDP/QUIC.
11. Добавить bypass rules.
12. Добавить диагностику и hardening.

## Что не делать в первой версии

Не стоит тащить в MVP:

- iOS parity;
- desktop support;
- собственный сетевой стек с нуля;
- MITM для HTTPS;
- слишком много proxy protocols сразу;
- удаленную панель управления конфигами;
- сложную multi-profile систему до стабилизации базового сценария.

## Ближайшие технические задачи

Первый реалистичный набор задач для старта разработки:

1. Создать Flutter-модели `ProxyConfig`, `InstalledApp`, `VpnStatus`.
2. Сделать Android plugin/channel для получения списка приложений.
3. Сделать экран выбора приложений.
4. Сделать экран настроек `SOCKS5`.
5. Реализовать `prepareVpn` и пустой `VpnService`.
6. Ограничить VPN только выбранными package names.
7. Добавить базовый экран статуса и кнопку остановки.

## Решения, которые стоит принять заранее

Перед кодом желательно окончательно выбрать:

- какое сетевое ядро встраиваем;
- какой минимальный набор proxy types поддерживаем в MVP;
- как храним секреты;
- нужен ли UDP уже в первой публичной версии;
- нужен ли один профиль или сразу несколько.
